## 1. 项目背景与目标

### 1.1 背景
传统日历工具（系统日历、办公日历）的核心录入方式仍是「点开 App → 选日期 → 填标题 → 设时间 → 设提醒 → 保存」，平均一次新建事件需要 6~9 次点击、20~40 秒。这种高摩擦的录入方式导致：

- 用户在走路、开车、开会间隙等「双手不空」场景下无法快速记录；
- 临时安排（「明天下午三点开会」）常常因为录入麻烦而被遗忘；
- 改期 / 删除等操作更繁琐，用户干脆放弃维护，日历逐渐失真。

语音是人类表达时间安排最自然的方式。一句「我明天下午三点到五点有个会议」包含了标题、日期、起止时间等全部要素，远比层层点选高效。

### 1.2 产品目标
打造一个**以语音交互为核心**的日历管理工具，让用户「开口即记、说完即成」，把单次事件录入从 30 秒降到 5 秒内，并能用语音完成查看、修改、删除、提醒等全生命周期管理。

### 1.3 衡量指标（北极星 & 关键指标）
| 指标 | 目标值 | 说明 |
| --- | --- | --- |
| 语音录入成功率 | ≥ 92% | 一句话被正确解析并落库的比例 |
| 单次录入耗时 | ≤ 5s | 从按下录音到事件确认入库 |
| 语义解析准确率 | ≥ 90% | LLM 解析出的结构化字段与用户真实意图一致 |
| 7 日留存 | ≥ 35% | 衡量是否形成记录习惯 |
| 语音录入占比 | ≥ 70% | 语音 vs 手动录入，验证「语音为核心」是否成立 |

## 2. 用户研究

### 2.1 目标用户画像
| 画像 | 典型场景 | 核心诉求 |
| --- | --- | --- |
| 职场白领（核心） | 会议多、临时安排多、通勤途中 | 快速记会议、提醒别迟到、避免冲突 |
| 销售 / 外勤 | 双手忙、频繁拜访客户 | 边走边记、改期方便 |
| 学生 | 课程、作业 DDL、考试 | 录入快、重复事件、临近提醒 |
| 长辈用户 | 不擅长打字操作 | 说话就能记，界面简单 |

### 2.2 真实需求挖掘（来自用户访谈/痛点归纳）
1. **录入要快**：用户最反感「为了记一件事点一堆按钮」。
2. **听得懂人话**：要能理解「明天」「下周一」「下午三点」「半小时后」「每周例会」等自然表达。
3. **要先确认再保存**：语音难免误识别，用户希望「先看一眼解析结果，对了再保存」，避免脏数据。
4. **冲突要提醒**：新事件和已有事件时间重叠时要主动告警。
5. **改 / 删要简单**：「把明天的会改到四点」「取消今天下午的牙医」也应能一句话完成。
6. **查询要自然**：「我明天有什么安排？」「这周有几个会？」。
7. **提醒要靠谱**：到点必须提醒，可自定义提前量。
8. **隐私安心**：语音内容、日程数据的存储与使用要透明。

### 2.3 竞品对照（差异化）
| 维度 | 系统日历 | 主流办公日历 | 语音助手（Siri等） | **VoiceCal** |
| --- | --- | --- | --- | --- |
| 语音录入 | 无/弱 | 弱 | 有但浅 | **核心能力，深度优化** |
| 自然语言解析 | 弱 | 中 | 中 | **LLM 深度解析，多要素一次到位** |
| 先确认后保存 | — | — | 无 | **有，可二次语音修正** |
| 冲突检测 | 有 | 有 | 无 | **有，并语音播报** |
| 改/删语音化 | 无 | 无 | 弱 | **支持一句话改期/删除** |

## 3. 产品定位与核心价值

**一句话定位**：开口即记的语音日历。

**核心价值闭环**：
> 说出来（语音输入）→ 听得懂（LLM 解析为结构化意图）→ 看得准（解析结果卡片确认）→ 存得对（服务端校验入库）→ 提醒到（到点通知）。

**核心交互主张**：首页中心是一颗醒目的**语音按钮**，长按或单击即说，所有高频操作（增/删/查/改）都能从这颗按钮发起。

## 5. 详细功能需求

下面对核心功能展开描述，给出「触发方式 → 输入 → 处理 → 输出 → 异常」。

### 5.1 F-VOICE-01 语音录音与转写
- **触发**：首页中央语音按钮（单击开始/结束，或长按说话松手结束）。
- **输入**：用户语音。
- **处理**：录音 → ASR 语音转文字（端上或云端）→ 文本上传后端解析。
- **输出**：界面实时显示识别出的文字（如「我明天下午三点到五点有个会议」）。
- **异常**：无麦克风权限 → 引导开启；识别为空 → 提示「没听清，请再说一次」并保留文本框可手动补充。

### 5.2 F-VOICE-03 解析结果确认卡（关键体验）
- **触发**：语音/文本解析完成后自动弹出。
- **输入**：LLM 返回的结构化 JSON。
- **处理**：把 JSON 渲染为可读卡片，对识别置信度低的字段做高亮（黄色）提示用户核对。
- **输出**：卡片展示「标题 / 日期 / 开始 / 结束 / 地点 / 提醒 / 重复」，提供【确认保存】【修改】【重说】【取消】四个动作。
- **价值**：解决语音误识别带来的脏数据问题，是「准确、顺畅」的关键保障。

### 5.3 F-ADD-01 语音添加单个事件
- **示例**："我明天下午三点到五点有个会议，地点在 3 号会议室，提前 15 分钟提醒"。
- **解析目标字段**：
  - intent = `create`
  - title = 会议
  - date = 明天（解析为绝对日期）
  - start_time = 15:00，end_time = 17:00
  - location = 3 号会议室
  - reminder = 15 分钟前
- **处理**：服务端校验（时间合法、是否冲突 F-CONFLICT-01）→ 写库 → 返回事件 ID。
- **输出**：确认卡 → 保存成功 toast → 今日/对应日视图刷新出现该事件。

### 5.4 F-VIEW-03 语音查询日程
- **示例**："我明天有什么安排？"
- **解析**：intent = `query`，time_range = 明天全天。
- **输出**：语音播报 + 列表展示当天事件；无安排时回复「明天暂无安排」。

### 5.5 F-EDIT-01 语音修改事件
- **示例**："把明天下午的会议改到四点开始"。
- **解析**：intent = `update`，target（用「明天+会议」定位事件），patch = { start_time: 16:00 }。
- **处理**：若命中多条 → 弹出歧义选择列表让用户点选；命中唯一 → 走确认卡 → 更新。
- **异常**：未命中事件 → 提示「没有找到匹配的日程」。

### 5.6 F-DEL-01 语音删除事件
- **示例**："取消今天下午的牙医预约"。
- **解析**：intent = `delete`，target = 今天 + 牙医。
- **处理**：命中后确认卡二次确认（防误删）→ 删除 → 支持 5 秒内撤销。

### 5.7 F-CONFLICT-01 时间冲突检测
- **触发**：create / update 落库前。
- **处理**：查询同时间段已有事件，若重叠则返回 conflict 列表。
- **输出**：确认卡顶部红色提示「与『xx 会议（14:30-15:30）』时间冲突」，用户可选择【仍然保存】或【调整时间】。

### 5.8 F-REMIND-01 / 02 提醒
- 到达「事件开始时间 − 提前量」时触发通知；通知点击进入事件详情。
- 默认提前量可在设置中配置（F-SET-01），单个事件可覆盖默认值。

## 6. 核心业务流程

### 6.1 语音添加事件主流程
```
用户长按语音按钮说话
        │  (F-VOICE-01)
        ▼
ASR 转文字 ──► 前端展示识别文本
        │
        ▼  POST /api/voice/parse  (原始文本)
   后端调用 LLM 解析  (见 02 文档 JSON Schema)
        │
        ▼  返回结构化 JSON (intent=create, 各字段)
   前端渲染「解析结果确认卡」 (F-VOICE-03)
        │
   ┌────┴─────────────┐
   │ 用户确认           │ 用户修改/重说
   ▼                    ▼
POST /api/events    回到语音/编辑 (F-VOICE-04 / F-EDIT-02)
   │
   ▼ 服务端校验 + 冲突检测 (F-CONFLICT-01)
   ├── 有冲突 ─► 确认卡提示，用户决定
   └── 无冲突 ─► 写入数据库
        │
        ▼
   返回成功 ─► 视图刷新 + 注册提醒 (F-REMIND-01)
```

### 6.2 意图分发
后端根据 LLM 返回的 `intent` 分发：`create / query / update / delete / unknown`。`unknown` 时引导用户换种说法或手动操作。

## 9. 异常与边界情况
| 场景 | 处理 |
| --- | --- |
| 识别为空/听不清 | 提示重说，保留文本框可补充 |
| 时间表达模糊（"晚点"/"过两天"） | 解析为待确认，确认卡高亮让用户补全 |
| 只说了标题没说时间 | 默认给出建议时间并高亮，让用户确认或重说 |
| 修改/删除命中多条 | 弹歧义列表让用户选择 |
| 修改/删除命中 0 条 | 提示未找到 |
| 跨天事件（23:00-次日01:00） | end 自动归到次日 |
| 时间冲突 | F-CONFLICT-01 流程 |
| 无网络 | 文本/语音转写失败提示，已有日程可离线查看（缓存） |

## 11. 验收标准（节选）
- 说「我明天下午三点到五点有个会议」→ 确认卡正确显示明天 15:00-17:00 标题「会议」→ 保存后出现在对应日视图。
- 说「我明天有什么安排」→ 正确列出/播报明天事件。
- 说「把明天的会改到四点」→ 命中事件并更新开始时间为 16:00。
- 说「取消今天下午的牙医」→ 命中并删除，5 秒内可撤销。
- 新增与已有事件重叠 → 确认卡出现冲突提示。
